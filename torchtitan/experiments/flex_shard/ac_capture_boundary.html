FlexShard Activation Checkpoint Capture and Prefetch

This note explains how FlexShard addresses the activation checkpointing requirement that full-parameter construction and all full-parameter uses must be replayable together, while still preserving backward all-gather prefetch across transformer layers.

Example setup

The examples below use a transformer with one explicit bucket per layer. Each bucket resolves to the layer module whose forward covers that bucket's parameter reads: 

BucketSpec(["layers.0.*"], reshard_after_forward=True) -> model.layers[0]
BucketSpec(["layers.1.*"], reshard_after_forward=True) -> model.layers[1]

1. Capture boundary and forward prefetch

Problem

To free full parameters via activation checkpointing, the replayed region must contain the dataflow from parameter all-gather to every use of the resulting full parameter: 

parameter all-gather -> full parameter tensor -> every full-parameter use

Forward prefetch makes this harder. If an all-gather is launched outside the checkpointed region that consumes the parameter, then activation checkpointing no longer owns the full producer-to-user chain cleanly.

FlexShard solution

Prefetch produces only a pending all-gather handle. The current bucket's pre-forward hook still finishes or launches _BucketAllGather.apply inside the layer's activation checkpoint region, exposes full parameters to the layer's parameter accessors, and then the layer forward reads them.

That keeps the activation-checkpointed replay boundary around the operation that materializes full parameters and all parameter uses in the layer.

Forward example

AC region for layers.0 forward order
layers.0 pre-forward hook
all-gather current bucket: layers.0
prefetch next forward bucket: layers.1
layers.0.forward()
reads full layers.0.* parameters
AC region for layers.1 forward order
layers.1 pre-forward hook
consume prefetched current bucket: layers.1
layers.1.forward()
reads full layers.1.* parameters

2. Backward prefetch

Problem

Activation checkpointing alone would recompute the all-gather only when backward reaches the checkpointed layer: 

backward reaches layer
  -> activation checkpoint recomputes that layer's all-gather
  -> layer backward compute can run

That is correct, but it puts the all-gather on the critical path. It does not naturally preserve the usual backward all-gather prefetch schedule.

FlexShard solution

FlexShard keeps bucket scheduling state outside any one checkpointed layer. During recompute, the bucket hook detects reshard-after-forward recompute, reverses the bucket order, and prefetches the next bucket in backward order after materializing the current bucket.

The first recomputed bucket launches on demand. After that, each recompute hook moves the next all-gather earlier, so the next layer can consume a pending handle instead of launching from scratch on its own critical path.

Backward recompute example

AC recompute region for layers.1 reverse order
layers.1 pre-forward hook
all-gather current bucket on demand: layers.1
prefetch next backward bucket: layers.0
layers.1.forward() recompute
rereads full layers.1.* parameters
layers.1 backward compute
overlaps the prefetched layers.0 all-gather
AC recompute region for layers.0 reverse order
layers.0 pre-forward hook
consume prefetched current bucket: layers.0
layers.0.forward() recompute
rereads full layers.0.* parameters

Invalid bucket shape

If a bucket spans parameters whose uses are not replayed by the same forward module, FlexShard cannot rely on activation checkpointing to recreate full parameters correctly. For example, a root-level bucket that includes layers.1.* is unsafe if checkpoint recompute only calls layers.1, because the root hook would not rerun.

In that case, FlexShard rejects the bucket and asks the user to split it so the bucket hook lands on the module that activation checkpointing will replay.
